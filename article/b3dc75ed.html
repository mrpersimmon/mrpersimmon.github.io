  5. 生成一个 etcd 的 Pod YAML 文件，通过 Static Pod 方式启动 etcd。

kubeadm join 工作流程

为什么执行 kubeadm join 需要一个 token 呢？

用于在 kube-apiserver 注册。

待加入节点的 kubeadm 需要「非安全模式」访问一次 Master 节点 kube-apiserver，从而获取到保存在 ConfigMap 的 CA 文件。

bootstrap token 用于首次「非安全模式」访问时的安全验证。

拿到证书后，待加入节点就可以以「安全模式」连接到 apiserver 了，新节点部署完成。

第一个 Kubernetes 应用

创建一个应用的最基本操作

一个 YAML 文件，就是一个 API 对象。

kind 指定了这个 API 对象是一个 Deployment。

spec.replicas 指定 Pod 副本数为 2。

spec.template 定义了一个 Pod 模版，描述了想要创建的 Pod 细节。

spec.template.spec.containers.image 定义了 容器的镜像信息

Pod 是 Kubernetes 世界里的「应用运行单元」，而一个应用运行单元可以由多个容器组成。

「控制器模式」：一种 API 对象（Deployment）管理另一种 API 对象（Pod）的方法。

metadata 是 API 对象的「标识」，即元数据，是从 Kubernetes 里找到这个对象的主要依据，用到的最主要的字段就是 metadata.labels。

spec.selector.matchLabels 使用一组键值对格式的标签定义过滤规则（Label Selector）。比如：app: nginx。Deployment 会把所有正在运行的、携带 app: nginx 标签的 Pod 识别为被管理的对象，并确保这些 Pod 的总数严格等于 2.

metadata 中还有一个 metadata.annotations，专门用来携带键值对格式的内部信息（对这些信息感兴趣的是 Kubernetes 组件，而不是用户）。大多数会在运行过程中被自动加在 API 对象上。

一个 API 对象的定义，大多可以分为 Metadata 和 Spec 两个部分。

  • Metadata：存放对象的元数据，对所有 API 对象来说，这部分字段和格式基本相同；
  • Spec：对象独有定义，用来描述所要表达的功能。

给这个 Deployment 加一个 Volume 声明

Volume 属于 Pod 对象的一部分。所以，需要修改 template.spec 字段。

添加了一个 volumes 字段，定义了这个 Pod 声明的所有 Volume。名字叫 nginx-vol，类型是 emptyDir。

什么是 emptyDir 类型呢？

等同于 Docker 的隐式 Volume 参数，不显示声明宿主机目录的 Volume。Kubernetes 会在宿主机上创建一个临时目录，这个目录将会被绑定挂载到容器声明的 Volume 目录上。

Pod 中的容器使用 volumeMounts 字段来声明自己要挂在哪个 Volume，并通过 mountPath 字段来定义容器内的 Volume 目录，比如 /usr/share/nginx/html。

小结

  • 使用一个 YAML 文件来描述要部署的 API 对象；然后统一使用 kubectl apply 命令完成对象的创建和更新操作。

  • 「最小」的 API 对象是 Pod。Pod 可以等价为一个应用，可以由多个紧密协作的容器组成。

  • 控制器模式：通过一种 API 对象来管理另一种 API 对象很常见。如 Deployment 控制 Pod，因为 Pod 是「最小」的对象，它往往被其他对象控制。这种组合方式是 Kubernetes 进行容器编排的重要模式。

  • API 对象，往往由 Metadata 和 Spec 两部分组成。其中，Metadata 里的 Labels 字段是 Kubernetes 过滤对象的主要手段。在这些字段中，容器想要使用的 Volume 正是 Pod 的 Spec 字段的一部分。而 Pod 里的每个容器，则需要显式声明自己要挂在哪个 Volume。

  • 务必要从命令行操作，向 kubectl apply YAML 文件 这种声明式 API 转变。

  • 快速熟悉 Kubernetes，联系流程

    • 在本地通过 Docker 测试代码，制作镜像；
    • 选择合适的 Kubernetes API 对象，编写对应 YAML 文件（如 Pod、Deployment）；
    • 在 Kubernetes 上部署这个 YAML 文件。

    应用部署后，接下来的所有操作，要么通过 kubectl 来执行，要么通过修改 YAML 文件来实现，不要再碰 Docker 命令行了。
